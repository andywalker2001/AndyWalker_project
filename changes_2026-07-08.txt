AIRPLANE SYMBOL ROTATION (Myfuncs.py — plot_plane / rotate_icon)
--------------------------------------------------------------------------------------
BUG FIX 1: PIL rotates counter-clockwise; aviation headings are clockwise.
  img.rotate(angle) was rotating icons in the wrong direction. Fixed by negating
  the angle: img.rotate(-angle).

BUG FIX 2: All aircraft shared a single rotated icon file (ac_icon_rotated.png).
  Each call to plot_plane overwrote the same file, so all Folium markers in the
  rendered HTML showed the last aircraft's heading. Fixed by returning a base64
  data URL from rotate_icon so each marker embedded its own unique image.

BUG FIX 3: Base icon (ac_icon.png) points NE (~45°), not North.
  The rotation formula img.rotate(-angle) assumed the base icon pointed North,
  causing a 45° offset error. Fixed with img.rotate(45 - angle).

REFACTOR: Replaced PIL/base64 approach with CSS DivIcon.
  The base64 PNG strings caused Jinja2 to hang during my_map.save() because it had
  to compile very large template strings. Replaced with a folium.DivIcon using the
  ✈ Unicode character and CSS transform:rotate(). This is instantaneous, requires no
  file I/O or image processing, and correctly respects the color parameter.
  Rotation formula for CSS (also clockwise): rotate(nav_heading - 45)deg.

TERRAIN MASKING OPTIMIZATION (AC_Range.py / Myfuncs.py)
--------------------------------------------------------------------------------------
IMPROVEMENT: Adaptive segment count based on 2D distance.
  get_masking previously used a hardcoded 5 segments regardless of path length.
  For a 100 km path that means one sample every 20 km, missing terrain features.
  For a 2 km path, 5 samples is wasteful. OpenMeteo returns ~1 km² resolution
  terrain data, so one sample per km is the optimal density. The segment count is
  now computed as max(1, int(distance_km)) using the 2D surface distance from
  calculate_slant_range[3]. All points are still sent in a single HTTP request to
  OpenMeteo, so there is no extra latency.

app_threaded.py — BACKGROUND TERRAIN MASKING (NEW FILE)
--------------------------------------------------------------------------------------
NEW: app_threaded.py (copy of app.py with background masking)
  Created an enhanced version of app.py that performs terrain masking in the
  background without blocking the /update response.

ARCHITECTURE: Two-phase map update for instant UI response.
  Phase 1 (immediate): Draws all aircraft in blue and saves the map. The /update
  route returns in ~1 second, so the browser sees aircraft appear instantly.
  
  Phase 2 (background): A daemon thread runs after the response is sent. It:
    - Calls get_masking() for each aircraft in parallel using ThreadPoolExecutor
    - Waits only for the slowest API call instead of running them sequentially
    - Redraws the map with red (terrain-masked) / blue (clear) colors
    - Saves the updated map file
  
  The browser's existing /map_ts polling (every 5 seconds) detects the file
  timestamp change and automatically reloads the map. The user sees aircraft
  recolor from blue to red/blue a few seconds after Update completes.

NEW GLOBAL: _map_lock (threading.Lock)
  Prevents race conditions if the user clicks Update again while a background
  masking thread is still writing the map file.

NEW GLOBAL: _masking_status dict
  Tracks whether background masking is running and when it last completed.
  Allows the UI to show optional status indicators.

NEW ENDPOINT: /masking_status
  Returns {"running": true/false, "last": "HH:MM:SS UTC"} so the frontend can
  display a "masking in progress..." message if desired.

NEW FUNCTION: _apply_masking_background()
  The background worker function that runs in a daemon thread. Uses
  ThreadPoolExecutor to parallelize get_masking() calls across all aircraft,
  dramatically reducing latency compared to sequential execution.

KEY DIFFERENCES from app.py:
  1. app.py skips terrain masking entirely (all aircraft blue) for speed
  2. app_threaded.py returns instantly with blue aircraft, then recolors in the
     background with actual masking results
  3. Masking API calls run in parallel instead of sequentially
  4. User never waits — the UI is always responsive
  5. Requires threading support (daemon threads, locks) not present in app.py
